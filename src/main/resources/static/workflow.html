URLViewer 업무 흐름 개요
🏆 IT개발실/IT카드개발팀(백명선, 김수현, 박귀렬 등)이 전체 프로세스를 주도하며, 분석 → 차단 확정 → 배포 → 모니터링까지 책임감을 갖고 이끌어 갑니다. 이 흐름은 1~3개월마다 반복 수행됩니다.
URLViewer는 시스템에 등록된 URL을 자동으로 수집·분석하고, 실제 사용 여부와 변경 이력을 종합하여 차단 대상 여부를 판정하는 내부 관리 도구입니다. 아래 흐름은 관리자(IT카드개발팀)가 수집·설정하고, 자동 처리가 주기적으로 데이터를 갱신하며, 판정 결과를 현업 검토자가 최종 확인하는 전체 업무 흐름을 보여줍니다.

3. 상태 판정 기준 (5가지 상태)
상태 판정
차단완료 차단완료
소스에 폐기 표시·차단 주석·차단 처리 코드가 모두 반영된 상태.
3가지 조건 모두 충족. 모든 수정 불가.
자동 · 변경불가
최우선 차단대상 최우선
실제 사용 이력 없음 + 1년 이상 미수정. 가장 시급히 차단을 검토해야 할 대상.
자동
후순위 차단대상 후순위
침해사고 이력·IT개발실 검토건 등 수동 지정 대상. 현업 협의 후 처리.
수동 지정
검토필요 차단대상 검토필요
사용 이력 없음 + 1년 미만 미수정 또는 사용 이력 있음 + 1년 이상 미수정. 추가 검토 필요.
자동
사용 사용
위 조건에 해당하지 않는 기본값. 정상 운영 중으로 판정.
기본값

PRIORITY 2

›
PRIORITY 3
@Description / Description 주석
메소드 위의 @Description 어노테이션 또는 /** @description */ 태그.
›
PRIORITY 4
컨트롤러 주석
컨트롤러 클래스 최상단 /** … */ JavaDoc 주석 첫 문장.
›
FALLBACK
빈 값
위 네 가지가 모두 비어 있을 때 표시값은 공란. 현황/상세 화면에서 직접 입력해 두면 이후 재분석에도 유지됩니다.
※ 분석현황 "내용" 컬럼 및 상세화면 "관련 메뉴(또는 기능)"에 표시되는 값은 위 우선순위 중 가장 먼저 값이 존재하는 항목을 선택합니다. 사용자 수정값(descriptionOverride)을 지우면 다시 자동 파싱 우선순위로 내려갑니다. 차단완료 상태는 수정할 수 없습니다.

6. 차단 확정 · 배포 · 모니터링 전체 흐름
현업 검토결과를 메일로 취합하여 IT개발실이 일괄 업로드하고, SmartWay 결재 → 개발 · 배포 → 운영 모니터링까지 IT카드개발팀이 주도하는 전체 운영 흐름입니다.
분석 · 검토 단계
👤
IT개발실
STEP 1 · 차단 대상 확정
URL 분석현황 화면에서 차단 대상 URL을 필터링하고 상태를 확정합니다.
차단 대상 목록을 엑셀로 다운로드하여 현업 검토 요청용 자료로 활용합니다.
↓
📋
협조전
+ 메일
+ SmartWay
STEP 2 · 현업 검토 요청 / 회신
① IT개발실이 협조전을 상신하여 공식 검토 요청을 진행합니다.
② 엑셀(URL 목록)을 첨부하여 메일로 검토 요청을 추가 발송합니다.
③ 차단확정 대상은 SmartWay 티켓을 통해서도 현업 담당자에게 검토를 요청합니다.
현업 담당자 회신: 차단대상 제외 / 차단확정 / 판단불가
↓
👤
IT개발실
STEP 3 · 검토결과 엑셀 업로드
회신받은 검토결과를 엑셀에 정리한 뒤 현업 검토 또는 URL 분석현황 화면에서 엑셀 업로드합니다.
검토결과가 일괄 반영되어 각 URL의 상태에 적용됩니다.
↓
🔗
SmartWay
STEP 4 · SmartWay 티켓 발행
URL 분석현황 화면에서 대상 URL을 선택 후 SmartWay 발행 버튼을 클릭합니다.
SmartWay에 업무 티켓이 생성되며 티켓 번호가 자동으로 저장됩니다.
SmartWay에서 현업 담당자의 결재·승인이 진행됩니다.
↓
🔄
SmartWay
STEP 5 · SmartWay 결재 결과 동기화
SmartWay 결재 완료 후 SmartWay 동기화 버튼을 클릭하여 결재 결과를 URLViewer로 가져옵니다.
승인(Done) 또는 반려(Won't Do) 결과가 각 URL의 검토 단계에 자동으로 반영됩니다.
↓
👤
IT개발실
STEP 6 · 최종 상태 확정
URL 분석현황 화면에서 결재 결과를 최종 확인하고 일괄 확정합니다.
차단 대상 목록을 개발팀에 전달하여 소스 수정을 요청합니다.
개발 · 배포 단계
↓
💻🧪
개발팀
STEP 7 · 개발 및 테스트
차단 확정된 URL에 대해 아래 3가지 조건을 모두 소스에 반영합니다.
① @Deprecated 어노테이션 추가 (폐기 표시) ② [URL차단작업] 주석 삽입 (차단 근거 명시) ③ UnsupportedOperationException throw 처리 (실제 차단)
소스 수정 완료 후 단위 테스트 및 통합 검증을 수행합니다.
차단 URL 호출 시 정상 차단되는지, 기존 기능에 영향은 없는지 확인합니다.
↓
🚀
CBO 배포
STEP 8 · CBO 배포 약 1주
CBO 환경에 배포 후 약 1주간 운영하며 이상 여부를 확인합니다.
문제 발생 시 즉시 롤백하고 원인을 분석합니다. 이상 없으면 운영 배포를 진행합니다.
↓
✅
운영 배포
STEP 9 · 운영 배포
CBO 검증이 완료된 소스를 운영 환경에 배포합니다.
배포 완료 후 URLViewer 다음 소스 수집 시 해당 URL이 차단완료 상태로 자동 전환됩니다.
↓
📡
모니터링
STEP 10 · 배포 후 모니터링 약 1주
운영 배포 후 약 1주간 모니터링 시스템을 통해 차단된 URL로 실제 인입이 있는지 확인합니다.
인입이 감지되면 원인을 분석하여 즉시 대응합니다. 이상 없으면 차단 작업을 최종 완료로 처리합니다.
🔁
전체 흐름(STEP 1~10)은 1~3개월마다 반복 수행합니다. IT카드개발팀이 주기를 설정하고, 분석 → 검토 → 배포 → 모니터링 사이클을 지속적으로 운영합니다.

9. URL 차단 여부 판정 기준
소스 수집 시 각 URL에 대해 아래 순서대로 판정이 수행됩니다. 앞 단계에서 확정되면 이후 단계는 건너뜁니다.
1
이미 차단완료된 URL
소스에 차단 처리가 완료된 URL — 상태 변경 없이 건너뜁니다.
↓ 아니면
2
담당자가 상태를 확정한 URL
확정 처리된 URL — 기존 상태를 유지하며 자동 재판정에서 제외됩니다.
↓ 아니면
3
현업 검토 결과 '차단 제외'로 확정된 URL
'차단대상 제외'로 검토 완료된 URL — 사용 상태로 유지됩니다.
↓ 아니면 → 자동 판정 실행

10. URL 차단여부 플로우차트
소스 수집 시 각 URL 1건이 거치는 상태 판정 전체 흐름입니다. 차단완료 여부 → 상태확정(statusOverridden) 여부 → 현업검토결과(reviewResult) 순으로 우선 확인한 뒤, 해당하지 않는 경우에만 자동 조건 계산이 실행됩니다.

11. 내용(관련 메뉴/기능) 판정 플로우차트
분석현황 "내용" 컬럼 및 상세화면 "관련 메뉴(또는 기능)"에 표시되는 값을 결정하는 우선순위 판정 흐름입니다. 사용자 수정값(descriptionOverride)이 최우선이며, 값이 없으면 소스 파싱값을 단계별로 내려가며 확인합니다. 가장 먼저 값이 존재하는 항목이 선택되고, 모두 비어 있으면 공란으로 표시됩니다.
