망분리 환경에서 현업 접근성 확보 · 2단계 게이트 설계
1배경 · 제약 조건
URLViewer는 시스템망에서 동작하며 개발자와 IT 담당자만 접근 가능합니다. 차단 대상 URL을 실제로 사용하는 현업 담당자는 업무망에서만 일하기 때문에 자동 탐지된 차단 후보에 대한 공식 의사결정 채널이 부재한 상태입니다.
URLViewer 위치
시스템망에 위치. 현업이 직접 접근 불가.
제약
업무망 이전 불가
정보보호부서 반대. 시스템망 유지 필수.
제약
메일 발송 불가
현업으로의 자동 알림 채널 부재.
제약
Jira는 업무망 존재
현업이 일상적으로 사용. 유일한 망간 브리지.

8차단 작업 반영
@Deprecated + 주석 + UnsupportedOperationException 3종 세트 반영. BLOCKED 상태로 종료.
🔒 차단 완료
4검토 단계 모델 (reviewStage 7단계)
INTERNAL_PENDING URLViewer
자동 탐지됨. IT 담당자 1차 검토 대기 상태.
INTERNAL_CONFIRMED URLViewer
IT 담당자가 차단 대상으로 확정. Jira 발행 후보.
INTERNAL_EXCLUDED URLViewer
IT 담당자가 오탐·보류로 제외. 재발 시 자동 제외 후보.
JIRA_ISSUED Jira
Jira 티켓 발행됨. 현업 검토 진행 중.
JIRA_APPROVED Jira
현업 승인. 차단 작업 대기열 진입.
JIRA_REJECTED Jira
현업이 차단 제외 결정. 사용 상태로 복귀.
BLOCKED 공통
코드 차단 작업 완료. 라이프사이클 종료.
5DB 스키마 변경 (api_record 확장)
-- api_record 테이블에 6개 컬럼 추가
review_stage          VARCHAR(30)   -- 7단계 검토 상태 (INTERNAL_PENDING 등)
internal_reviewer     VARCHAR(50)   -- 1차 검토한 IT 담당자 사번/이름
internal_reviewed_at  TIMESTAMP     -- 1차 검토 완료 시각
internal_memo         TEXT          -- IT 담당자 메모 (오탐 사유 등)
jira_issue_key        VARCHAR(50)   -- Jira 티켓 키 (ex: URLB-1234)
jira_synced_at        TIMESTAMP     -- 마지막 Jira 동기화 시각
6URLViewer /review.html 리뉴얼
현업 검토가 붙기 전에도 IT 담당자의 1차 게이트가 효과를 내려면 /review.html을 대량 처리 특화 화면으로 보강해야 합니다.
  • 벌크 액션 체크박스 — 행 다중 선택 후 확정 / 제외 / 보류 / Jira 발행을 한 번에 실행.
  • 필터 프리셋 — "내 팀 검토 대기건", "호출 0건 & 커밋 1년 이상", "최근 7일 신규 탐지" 등 자주 쓰는 조건을 원클릭으로.
  • 디버깅 링크 — Controller 파일 경로, Git 커밋 이력, 호출 추이 차트를 행에서 바로 열람.
  • 오탐 마킹 — 한 번 오탐으로 처리한 건은 다음 배치에서 자동 제외 후보로 내려 반복 피로 제거.
7Jira 동기화 설계
PUSH URLViewer → Jira
  • 관리자가 INTERNAL_CONFIRMED 건 선택 → "Jira 티켓 발행" 버튼.
  • Jira REST API /rest/api/2/issue/bulk로 Story 일괄 생성.
  • Epic = 레포 × 분기, Story = URL 1건 구조로 정리.
  • 멱등성: api_record.id를 Jira 커스텀 필드 externalId로 넣어 중복 방지.
  • 발행 성공 시 jira_issue_key / jira_synced_at 업데이트, review_stage = JIRA_ISSUED.
PULL Jira → URLViewer
  • Quartz Job 1시간 주기로 Jira 폴링.
  • JQL: project=URLB AND updated > last_sync.
  • Jira Resolution 매핑 → reviewStage 전이
    · Done → JIRA_APPROVED
    · Won't Do → JIRA_REJECTED
    · Cannot Reproduce → 판단불가
  • 동기화 결과 jira_synced_at 갱신 · 감사 로그 기록.

10도입 로드맵
# 항목 비고
1 review_stage 컬럼 + 7단계 모델 도입 모든 후속 작업의 기반. 기존 status 체계와 병행 운영.
2 /review.html 벌크 액션 UI Jira 연동 없이도 즉시 효과. IT 담당자 생산성 향상 체감.
3 Jira REST 연동 PoC 망 정책(시스템망→업무망 API 호출 허용) 확인 필요.
4 Jira 발행 배치 + 역동기화 Job PoC 성공 시. PUSH + PULL 양방향 완성.
5 대시보드 진척률 카드 UI 개선. 관리자·경영진 가시성 확보.
1·2번만 먼저 해도 Jira 연동 없이 IT 담당자 검토 효율이 크게 오릅니다. Jira 연동은 네트워크 정책 확인 후 단계적으로 붙여도 충분합니다.
11확인 필요한 제약사항
  • 시스템망 → 업무망 Jira REST API 호출 가능 여부 (방화벽·프록시 정책)
  • Jira 프로젝트 신설 권한 및 커스텀 필드 추가 가능 여부 (externalId 필드)
  • Jira 서비스 계정 API 토큰 발급 가능 여부
  • 현업 계정 ↔ manager_mappings 매칭 키 (사번 등 유일 식별자)
  • 예상 티켓 규모 추정 (레포 수 × 평균 차단 대상 건수)로 Jira 라이선스·부하 검토
